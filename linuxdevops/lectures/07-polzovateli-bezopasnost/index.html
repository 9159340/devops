    7. Пользователи, доступ, безопасность

    Подробнее по краткой лекции и плану изучения.

    Учётные записи

    Файл Содержимое
    /etc/passwd Имя, UID, GID, домашний каталог, shell (пароль здесь не хранится)
    /etc/shadow Хеш пароля (только root)
    /etc/group Группы и их члены

    UID (User ID) / GID (Group ID) — числовые идентификаторы.

    • UID < 1000 (или 999) — системные учётки (root=0, nginx, sshd).
    • UID ≥ 1000 — обычные пользователи.
    id
id alice
getent passwd alice
getent group docker

    # Создание пользователя
sudo useradd -m -s /bin/bash alice
sudo passwd alice
sudo usermod -aG docker alice    # добавить в группу

    sudo и su

    sudo (superuser do) — выполнить команду от другого пользователя (обычно root) с записью в аудит.

    su (switch user) — сменить пользователя; нужен пароль целевого пользователя.

    sudo whoami
sudo -u postgres psql
su - alice                # login shell целевого пользователя

    Настройка sudo — только через visudo (проверяет синтаксис):

    sudo visudo
# или отдельные файлы:
sudo visudo -f /etc/sudoers.d/alice

    Пример правила:

    alice ALL=(ALL) NOPASSWD: /bin/systemctl restart nginx

    PAM (обзорно)

    PAM (Pluggable Authentication Modules) — стек проверки аутентификации: пароль, LDAP, 2FA, политики смены пароля. Конфиги в /etc/pam.d/ (отдельный файл на сервис: login, sshd, sudo).

    Для DevOps достаточно знать: если «не пускает» при правильном пароле — смотреть /etc/pam.d/ и логи.

    SSH

    SSH (Secure Shell) — протокол и набор утилит для удалённого защищённого доступа: интерактивная оболочка, выполнение команд, копирование файлов (scp, sftp), туннели. По умолчанию слушает порт 22 (TCP).

    Схема аутентификации по ключу:

    клиент (приватный ключ)  ←→  сервер (публичный ключ в authorized_keys)

    Генерация ключа: ssh-keygen

    Создаёт пару ключей: приватный (хранится только у владельца) и публичный (можно копировать на серверы).

    ssh-keygen -t ed25519 -C "alice@laptop"
    Операнд / опция Значение
    ssh-keygen Утилита генерации и управления ключами OpenSSH
    -t ed25519 -t (type) — алгоритм ключа. ed25519 — современный, короткий и быстрый (рекомендуется). Альтернативы: rsa (обычно -b 4096), ecdsa
    -C "alice@laptop" -C (comment) — комментарий в конце публичного ключа; удобно для идентификации («чей это ключ и с какой машины»). Не влияет на криптографию

    Интерактивные вопросы при запуске:

    Вопрос Что означает
    Enter file in which to save the key Путь к файлу приватного ключа. По умолчанию ~/.ssh/id_ed25519; Enter — принять значение по умолчанию
    Enter passphrase Дополнительный пароль на ключ. Пустой Enter — ключ без пароля (удобно для CI, менее безопасно на ноутбуке). С паролем — при каждом использовании нужна расшифровка (или ssh-agent)

    Результат:

    Файл Назначение Права
    ~/.ssh/id_ed25519 Приватный ключ — никому не отдавать 600 (chmod 600)
    ~/.ssh/id_ed25519.pub Публичный ключ — копируется на сервер 644

    Полезные дополнительные опции ssh-keygen:

    ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_work -C "alice@work"   # свой путь к файлу
ssh-keygen -p -f ~/.ssh/id_ed25519                                   # сменить passphrase
ssh-keygen -lf ~/.ssh/id_ed25519.pub                                   # показать отпечаток (fingerprint) ключа
    Опция Значение
    -f путь Явно задать файл приватного ключа
    -p Изменить passphrase существующего ключа
    -lf файл.pub Показать fingerprint публичного ключа

    Копирование ключа на сервер: ssh-copy-id

    Добавляет ваш публичный ключ в ~/.ssh/authorized_keys на удалённой машине.

    ssh-copy-id alice@server.example.com
    Операнд Значение
    ssh-copy-id Скрипт-обёртка: подключается по SSH, создаёт ~/.ssh, дописывает ключ в authorized_keys, выставляет права
    alice@server.example.com user@host — пользователь на сервере и адрес (DNS-имя или IP). Порт по умолчанию 22

    При первом подключении спросит пароль пользователя alice (если ещё включена парольная аутентификация). После этого вход по ключу должен работать без пароля.

    Варианты с нестандартным ключом или портом:

    ssh-copy-id -i ~/.ssh/id_ed25519_work.pub alice@server.example.com
ssh-copy-id -p 2222 alice@server.example.com
    Опция Значение
    -i файл.pub Явно указать публичный ключ (если не стандартный ~/.ssh/id_ed25519.pub / id_rsa.pub)
    -p 2222 Порт SSH на сервере (если не 22)

    На сервере в ~/.ssh/authorized_keys каждая строка — один разрешённый публичный ключ. Права: каталог ~/.ssh — 700, файл authorized_keys — 600.

    Подключение: ssh

    ssh alice@server.example.com
    Операнд Значение
    ssh Клиент SSH: открывает удалённую сессию (shell) или выполняет одну команду
    alice Имя пользователя на удалённой системе (логин)
    @ Разделитель «пользователь / хост»
    server.example.com Имя хоста или IP-адрес сервера

    Частые опции:

    ssh -p 2222 alice@server.example.com              # нестандартный порт
ssh -i ~/.ssh/id_ed25519_work alice@server       # конкретный приватный ключ
ssh -l bob server.example.com                    # то же, что bob@server (логин через -l)
ssh alice@server 'hostname; uptime'              # выполнить команду без интерактивного shell
ssh -v alice@server                              # v = verbose. подробный лог (отладка)
    Опция Значение
    -p порт TCP-порт SSH на сервере
    -i файл Путь к приватному ключу
    -l user Логин (альтернатива user@host)
    'команда' Выполнить одну команду и выйти
    -v / -vv / -vvv Уровень отладки (больше букв — больше деталей)

    Файл ~/.ssh/config на клиенте позволяет не писать опции каждый раз:

    Host myserver
    HostName server.example.com
    User alice
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_work

    После этого: ssh myserver.

    Hardening сервера: /etc/ssh/sshd_config

    sshd — демон SSH на сервере; конфиг — /etc/ssh/sshd_config.

    PermitRootLogin no
PasswordAuthentication no
Port 2222
    Директива Значение
    PermitRootLogin no Запретить прямой вход под root по SSH (рекомендуется: сначала обычный пользователь, затем sudo)
    PasswordAuthentication no Отключить вход по паролю; только ключи (или другие методы из AuthenticationMethods)
    Port 2222 Слушать порт 2222 вместо 22 (не замена firewall, но снижает шум от ботов)

    Другие полезные директивы:

    Директива Значение
    PubkeyAuthentication yes Разрешить вход по ключу (обычно по умолчанию)
    AllowUsers alice bob Белый список пользователей, которым разрешён SSH
    MaxAuthTries 3 Лимит попыток аутентификации за одно подключение

    Проверка и применение конфига:

    sudo sshd -t
sudo systemctl reload sshd
    Команда Операнды Значение
    sshd -t -t (test) Проверить синтаксис sshd_config без перезапуска. При ошибке — код выхода ≠ 0 и сообщение в stderr
    systemctl reload sshd reload Перечитать конфиг без обрыва уже открытых сессий. Имя юнита на Debian/Ubuntu часто ssh, на RHEL — sshd (systemctl reload ssh)

    После смены порта или отключения пароля держите вторую сессию открытой, пока не проверите новое подключение.

    Подключение через бастион: ProxyJump

    Bastion (бастионный хост или сервер-посредник) в контексте SSH — это выделенный, защищенный сервер, который служит единой точкой входа для подключения к внутренней, закрытой сети организации.
    Он действует как шлюз: администратор сначала подключается по SSH к бастиону, а затем с него — к нужным внутренним серверам.

    Чаще всего бастионы используются для следующих задач:
    - Усиленная безопасность: Внутренние серверы не имеют прямого доступа из интернета.
    Бастион закрывает их от внешних сканирований и атак.
    - Централизованный аудит: Все подключения проходят через один сервер, что позволяет легко собирать и анализировать централизованные логи того, кто и когда заходил в инфраструктуру.
    - Управление ключами: Вместо того чтобы администрировать SSH-ключи пользователей на десятках внутренних серверов, достаточно настроить доступ только на узле-бастионе.

    Для удобной работы через такой шлюз в SSH применяются специальные методы:
    - ProxyJump (переходной прыжок): Современный синтаксис SSH, который позволяет перенаправлять трафик через бастион без необходимости вручную открывать соединение до него, а затем со второй машины до финальной.
    Настраивается аргументом -J или в конфигурационном файле.
    - SSH-туннелирование: Использование локального перенаправления портов для создания защищенного туннеля через узел-бастион, например, для доступа к внутренним базам данных.
    - VPN-over-SSH: Некоторые утилиты (например, sshuttle) позволяют создавать VPN-подобные шифрованные соединения через SSH-бастион для доступа ко всей внутренней подсети.

    ProxyJump — цепочка: клиент → промежуточный хост (бастион) → целевой сервер во внутренней сети.

    ssh -J bastion.example.com target.internal
    Операнд / опция Значение
    -J bastion.example.com -J (jump host) — хост, через который проксируется соединение. Можно несколько: -J jump1,jump2
    target.internal Конечный хост (доступен с бастиона, не обязательно с вашего ноутбука)

    Эквивалент в ~/.ssh/config:

    Host target
    HostName target.internal
    User alice
    ProxyJump bastion.example.com

    Расширенный вариант — ProxyCommand (старый способ, гибче, но сложнее):

    ssh -o ProxyCommand="ssh -W %h:%p bastion.example.com" target.internal
    Плейсхолдер Значение
    %h Имя целевого хоста
    %p Порт целевого хоста

    SSH agent: ssh-agent и ssh-add

    ssh-agent — фоновый процесс, хранящий расшифрованные приватные ключи в памяти сессии. Нужен, если на ключе стоит passphrase: вводите пароль один раз, дальше ssh / git используют агент.

    eval "$(ssh-agent -s)"          # запустить агент в текущей shell-сессии
ssh-add ~/.ssh/id_ed25519       # добавить ключ в агент (спросит passphrase)
ssh-add -l                      # список загруженных ключей (fingerprints)
ssh-add -D                      # удалить все ключи из агента
    Команда / опция Значение
    ssh-agent -s Запуск агента; выводит переменные SSH_AUTH_SOCK, SSH_AGENT_PID для eval
    ssh-add файл Загрузить приватный ключ в агент
    ssh-add -l list — показать отпечатки ключей в агенте
    ssh-add -D delete all — очистить агент

    Передача агента на удалённый хост (осторожно: доверяйте только своим серверам):

    ssh -A alice@server.example.com    # -A — ForwardAgent yes

    Копирование файлов (кратко)

    scp file.txt alice@server:/tmp/                    # копия на сервер
scp -r ./local_dir alice@server:/opt/              # -r — рекурсивно (каталог)
scp alice@server:/var/log/syslog ./logs/           # скачать с сервера
sftp alice@server                                  # интерактивный SFTP (put, get, ls)
    Команда Ключевые опции
    scp -r — каталоги; -P порт — порт (заглавная P); -i — ключ
    sftp Те же -P, -i; внутри сессии: put, get, cd, lcd

    Linux capabilities

    Вместо полного root процессу можно выдать отдельные capabilities (возможности):

    Capability Зачем
    CAP_NET_BIND_SERVICE Слушать порт < 1024 без root
    CAP_SYS_ADMIN Администрирование (опасно)
    CAP_NET_RAW raw-сокеты (ping без root) 
    getcap /usr/bin/ping
sudo setcap cap_net_bind_service=+ep /opt/myapp/server

    Критично для контейнеров: Docker по умолчанию отбрасывает большинство capabilities.

    SELinux и AppArmor

    SELinux (Security-Enhanced Linux) — на RHEL/CentOS/Fedora. Режимы:

    • Enforcing — блокирует нарушения;
    • Permissive — только логирует;
    • Disabled — выключен.
    getenforce
sestatus
ls -Z /var/www/html/       # контекст безопасности
sudo restorecon -Rv /var/www/html/
getsebool -a | grep httpd

    AppArmor — профили для приложений на Ubuntu/Debian (aa-status).

    seccomp (обзорно)

    seccomp (secure computing mode) — фильтр syscall'ов: процессу разрешён только белый список системных вызовов. Docker и Kubernetes используют для уменьшения поверхности атаки контейнера.

    Практика

    1. Создать пользователя, выдать sudo только на один systemctl restart.
    2. Настроить вход по ключу, отключить PasswordAuthentication.
    3. getcap на /usr/bin/ping — увидеть CAP_NET_RAW.
